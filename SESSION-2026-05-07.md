# Session 2026-05-07 — P2P H.264 Video Pipeline

## Goals
- Implement end-to-end P2P video pipeline: PC1 encodes I420→H.264, sends via RTP, PC2 receives→decodes→renders
- Use gkit-media RTC API + webrtc-rs 0.17 backend + vnet for offline testing

## Accomplishments

### vnet ICE Connectivity (FIXED)
- Root cause 1: Missing `w.start()` on vnet Router — packets never forwarded
- Root cause 2: Dual tokio runtime (test's local `rt` vs global `rt()`) — vnet and ICE on separate event loops
- Fix: Unified runtime + `w.start()` → ICE now Connected

### on_track Callback (FIXED)
- Root cause: `create_video_track()` returns `WrtcVideoTrack` containing the frame generator
- The result was dropped immediately (`_ = create_video_track(...)` → `let _track = create_video_track(...)`)
- Fix: keep `_track` alive → generator thread continues, on_track fires

### H.264 RTP Payloader (BUG FOUND, WORKAROUND WIP)
- **rtp-0.17.1 H264Payloader bugs:**
  1. STAP-A (SPS+PPS aggregation) is NEVER created/pushed to payloads
  2. FU-A header has wrong NAL type (1 instead of 5 for IDR)
- **Encoder verified:** openh264 self-test decodes raw Annex B bitstream correctly (320x240 OK)
- **SPS/PPS correctly extracted** and stored in global `SPS_PPS_STORE`

### VP8 Passthrough (CURRENT APPROACH)
- Use VP8 mime type for codec negotiation (VP8 Payloader passes data through transparently)
- H.264 Annex B bitstream (SPS+PPS+IDR) written as VP8 payload
- VP8 Payloader fragments at MTU boundaries (~1150 bytes per chunk)
- Receiver: VP8 depacketizer strips descriptor, we reassemble chunks using S flag
- Each complete frame fed to openh264 decoder
- **Status:** Chunks arriving, VP8 descriptor parsing WIP, decoder still errors on incomplete data

### Attempted Approaches (all failed):
1. Direct H264 Payloader — STAP-A never created (bug)
2. Separate SPS/PPS + IDR write_sample calls — STAP-A still never created
3. Manual STAP-A + FU-A via TrackLocalStaticRTP::write_rtp() — packets sent but receiver never receives (H264 depacketizer incompatible)
4. Keyframe reassembly with corrected NAL type — IDR payload truncated

## Critical Learnings
- webrtc-rs 0.17 official tests all use VP8 codec — H.264 Payloader bugs untested by upstream
- `TrackLocal::write_rtp()` takes raw RTP packets but receiver's depacketizer may not handle them
- VP8 passthrough is the most viable approach for H.264 over webrtc-rs 0.17
- vnet requires `w.start()` AND all async operations on same tokio runtime
- `read_rtp()` returns RAW RTP payload (not depacketized) — need manual depacketization for H.264 FU-A
